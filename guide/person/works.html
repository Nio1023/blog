                    # 门店列表

                    • 技术栈：Vue2
                    • 通过数据驱动的 vue 模板渲染 DOM，通过第三方插件 html2canvas 生成图片并下载
                    • 大量节约设计师搬运表格信息生成图片的时间，最大程度的保证了图片信息的准确性
                    • 使得运营有机会有能力独立操作修改店铺内的图片错误
                    • 减少活动期间修改图片的时间，不必保留每个模块的 ps 文件
                    • 设计师原需 2-3 天制作图片，现在仅需 0.5 天，使得设计师有更充裕的时间和精力用于考虑页面排版和美观度上
                    • 运营原需提前 7 天提供产品表，现仅需提前 3 天，使得运营有更多时间选出更精确的产品

                    # 热区代码

                    • 技术栈：Vue2
                    • 通过处理表格信息生成多个热区，根据预设的图片宽度排列出整齐的热区布局，最终导出成 code 并 copy 到粘贴板上
                    • 节约上线人员所需的上线时间，使一个人同时负责多家店铺成为可能，原 2~3 天工时=> 0.5 天
                    • 通过对设计和上线的双重优化，大大缩短了天猫活动前期的准备时间，也为后面几年店铺面临越来越多的活动打下了基础

                    # 主图

                    • 技术栈：Vue2
                    • 根据产品 Code 获取产品信息，并以此为基础，创建了两个图片编辑平台，最后通过 html2canvas 生成 base64 后经 jszip 打包成图片压缩包
                    • 赋予运营主图和基础版本 PDP 的设计能力，并通过线上保存图片配置信息
                    • 通过在线编辑确保图片模板的一致性，避免不同人员手中的本地图片模板的版本不一致，最终的图片参差不齐
                    • 能最快的响应设计政策，调整设计模板
                    • PDP 设有埋点，上线以来累计登录 2157 人次，搜索产品 3697 次，生成详情页 2280 次，以外部供应商 180/个页面计算，保守估计仅 PDP 工具节约费用 20 万

                    # 电商跨部门协作平台

                    • 技术栈：Vue2
                    • 为不同的任务设置不同的任务流水线以及物料列表，规范交付内容，节省跨部门沟通成本，记录每个阶段的完成时间，设立交付阶段的审核关卡，允许流程回退，任务完成后归档支持 ID 查询
                    • 接入阿里云 OSS，在服务端封装 RAM 用户接口方便本地创建临时虚拟账户，实现素材的本地向 OSS 直传，减少对服务器带宽的影响
                    • 对于协同类平台，需要满足时效性，引入 websocket，为任务的实时提醒及实时更新等提供条件

                    # 天猫模块库

                    • 技术栈：vite + Vue3(多页面应用) + TypeScript
                    • 实现非大促级别活动的可定制化模块生成图片/Code，可复制模块/页面复用，提高活动上线的速度，保证了优秀模块/优秀活动的留存
                    • 通过 Campaign/Page/Module 三表分别存储信息，实现模块与页面解耦，页面与活动解耦
                    • 对常用的模块函数（cookie/localStorage/render 等）进行 hooks 封装，部分 hooks 使用单例设计模式封装
                    • 重新编写基于 TS 的方法函数，并为第三方插件创建本地描述文件
                    • 分析 vite2.x/rollup 的分包原理，解决了 chunk 碎片问题，并通过 CDN 方式引入 Element-plus 大大减少了网络请求
                    • 分析 html2canvas 的渲染原理，忽略大量编辑器的无关 dom，大大减少渲染所需时间（测试相同模块配置下：45s => 8s）
                    • 在服务端开启 Nginx 的 brotli 压缩，通过打包生成.br 的压缩文件，进一步压缩总包体积
